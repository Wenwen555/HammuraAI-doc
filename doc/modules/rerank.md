# 重排功能

## 模块目标

重排功能用于把检索召回的一批候选材料重新排序，让真正最相关、最适合当前法律任务的内容排到前面。

法律场景里，重排的价值不只是提升准确率，还在于减少错误引用和弱相关材料干扰。

## 当前实现

核心位置：

- `backend_app/app/services/workflow.py`
- `backend_app/app/services/prompts.py`

## 当前机制

### 轻量案例重排

当前 `workflow.py` 已实现一个简化版重排逻辑：

- 先召回参考案例
- 再按 `action` 增加轻量偏置
- 根据诉求方向做简单排序修正

这类逻辑适合作为第一阶段的基础版，因为：

- 成本低
- 行为可解释
- 方便快速验证

### 引用结构输出

当前系统已经会把重排后的结果转成：

- `citations`
- `rag_block.reference_cases`
- `rag_block.reference_provisions`

这意味着重排结果已经能进入前端摘要和 LLM 上下文，而不是停留在内部数组。

## PG + pgvector 下的融合排序

引入 pgvector 后，重排模块的职责会更清晰：数据库负责完成候选召回，重排模块负责把多路候选整理成适合生成层消费的证据包。

候选来源包括：

| 来源 | 排序信号 |
| --- | --- |
| pgvector Top-K | 语义相似度，适合召回表达方式不同但含义接近的材料。 |
| PostgreSQL 全文 / 关键词检索 | 词项匹配，适合案号、法条编号、专有名词和固定术语。 |
| 业务规则偏置 | 根据 `action`、用户问题类型、文档类型和资料新旧程度做轻量修正。 |

第一阶段可以采用轻量融合策略：

1. 标准化不同召回路径的分数。
2. 对相同 `chunk_id` 或相同引用位置的候选去重。
3. 优先保留命中法律依据明确、来源字段完整的片段。
4. 将最终 Top-K 写入 `citations` 与 `rag_block`。

概念结果结构：

```json
{
  "chunk_id": "chunk_001",
  "doc_title": "中华人民共和国民法典",
  "source_ref": "合同编 第五百七十七条",
  "vector_score": 0.82,
  "keyword_score": 0.46,
  "final_score": 0.76,
  "reason": "语义命中违约责任，并包含明确法条引用"
}
```

这类结构能让后续回答解释更稳，也能为评测集记录“为什么选中这条材料”。

## 第一阶段交付重点

1. 候选结果可排序。
2. 输出结果可附带引用信息。
3. `workflow` 与 `prompt` 能消费这些引用。
4. 多路召回结果能形成统一候选结构，避免生成层直接面对散乱索引结果。

## 当前不足

- 还没有独立的模型式 rerank 服务
- 规则偏置仍然较轻，适合 POC，不足以覆盖复杂法律问题
- 法规与案例的联合融合仍有优化空间

## 后续演进方向

- 引入真正的 rerank 模型
- 支持多路召回融合
- 输出更精细的可解释排序原因
- 在 pgvector 候选集基础上加入 RRF 或学习式重排
- 将重排输入和输出记录到审计日志，支撑法律问答质量复盘
